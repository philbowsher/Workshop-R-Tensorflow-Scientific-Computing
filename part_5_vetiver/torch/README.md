# Torch + Vetiver: Peptide Binding Prediction

Complete ML deployment pipeline using **Torch** and **Vetiver** for MLOps.

## Files

### 1_train_model.R
Train the peptide binding prediction model using torch (identical to part_4_torch training).

**Output:** `peptide_model_torch.pt`

---

### 2_version_model_vetiver.R
Create a Vetiver model object and pin to Connect.

**What it does:**
```r
# Load torch model
model <- torch_load("peptide_model_torch.pt")
model$eval()

# Create vetiver model with metadata
v <- vetiver_model(
  model = model,
  model_name = "peptide_torch_vetiver",
  description = "Peptide-MHC binding prediction (Torch)",
  metadata = list(
    framework = "torch",
    classes = c("NB", "WB", "SB"),
    accuracy = "~95%"
  )
)

# Pin to Connect (with automatic versioning)
vetiver_pin_write(board, v)
```

**What you get:**
- Model card with metadata
- Automatic versioning
- Torch model serialization handled automatically

---

### 3_deploy_api_vetiver.R
Deploy the API to Connect (automatically generated by Vetiver).

**What it does:**
```r
# This ONE line deploys the entire API!
vetiver_deploy_rsconnect(
  board = board,
  name = "peptide_torch_vetiver"
)
```

**What you get:**
- Auto-generated plumber.R
- Torch model loading handled automatically
- `/predict` endpoint
- `/ping` endpoint (health check)
- `/metadata` endpoint (model info)

**No manual plumber.R file needed!**

---

### 4_consume_api.R
Call the deployed Vetiver API.

**Key differences from manual API:**
- POST request (not GET)
- JSON body (not query params)
- Returns raw logits (not class labels)

**Example:**
```r
# Create endpoint
api_endpoint <- vetiver_endpoint("https://pub.workshop.posit.team/content/xyz-789")

# Prepare data (encoded peptides as data frame)
input_df <- as.data.frame(encoded_peptides)

# Make predictions
predictions <- predict(
  api_endpoint,
  input_df,
  httr::add_headers(Authorization = paste0("Key ", api_key))
)
```

## Torch-Specific Considerations

### Model Evaluation Mode
Vetiver automatically sets torch models to evaluation mode:
```r
# Vetiver does this for you:
model$eval()  # Disables dropout, batch norm, etc.
```

### No Gradient Computation
Vetiver wraps predictions in `with_no_grad()`:
```r
# Vetiver does this for you:
with_no_grad({
  predictions <- model(input)
})
```

### CPU vs GPU
Vetiver handles device placement automatically. If your model was trained on GPU, Vetiver will:
- Use GPU if available on Connect
- Fall back to CPU if not available

## API Endpoints

### POST /predict
Make predictions

**Request:**
```json
POST /predict
Authorization: Key your-api-key
Content-Type: application/json

{
  "V1": 0.5, "V2": 0.3, ..., "V180": 0.1
}
```

**Response (Torch returns logits):**
```json
[
  [-1.2, 2.5, 0.3],  // Peptide 1: [NB_logit, WB_logit, SB_logit]
  [-2.1, 0.8, 3.2]   // Peptide 2: [NB_logit, WB_logit, SB_logit]
]
```

**Note:** Torch models return logits (pre-softmax values), not probabilities. Apply softmax if needed:
```r
# Convert logits to probabilities
probs <- torch_softmax(torch_tensor(predictions), dim = 2)
```

## Keras vs Torch with Vetiver

### Same Workflow, Different Frameworks

| Step | Keras | Torch |
|------|-------|-------|
| Train | keras model | torch model |
| Create vetiver | `vetiver_model(keras_model, ...)` | `vetiver_model(torch_model, ...)` |
| Pin | `vetiver_pin_write(board, v)` | `vetiver_pin_write(board, v)` |
| Deploy | `vetiver_deploy_rsconnect(...)` | `vetiver_deploy_rsconnect(...)` |
| Consume | POST with data frame | POST with data frame |

### Differences in Predictions

**Keras:**
- Returns probabilities (post-softmax)
- Values sum to 1.0 per sample

**Torch:**
- Returns logits (pre-softmax)
- Need to apply softmax for probabilities

**Both work with Vetiver!** Choose based on your modeling needs.

## Troubleshooting

### Error: torch not found on Connect
**Solution:** Ensure torch is in Connect's R library. May need to contact admin.

### Error: Model predictions wrong shape
**Solution:** Ensure input is a data frame with 180 columns (V1-V180)

### Error: CUDA errors on Connect
**Solution:** Torch automatically falls back to CPU if GPU not available. Check Connect logs.

### Predictions are logits, need probabilities
**Solution:** Apply softmax:
```r
library(torch)
probs <- torch_softmax(torch_tensor(predictions), dim = 2)
probs_r <- as_array(probs)
```

## Production Tips

1. **Model Size**: Torch models can be large. Monitor storage on Connect.
   ```r
   # Check model size
   file.size("peptide_model_torch.pt") / 1024^2  # MB
   ```

2. **Inference Speed**: Torch can be faster than Keras for some operations
   ```r
   # Benchmark
   system.time({
     predictions <- model(x_test)
   })
   ```

3. **Batch Predictions**: Torch handles batches efficiently
   ```r
   # Predict 100 peptides at once
   input_df <- as.data.frame(encoded_100_peptides)
   predictions <- predict(api_endpoint, input_df)
   ```

4. **Memory Management**: Large models may need more RAM on Connect
   - Monitor Connect metrics
   - Consider model quantization for production
   - Use CPU inference if GPU unavailable

```r
# Train Torch model

# Deploy both
# ... (version and deploy both)

# Compare predictions on same peptides
keras_preds <- predict(keras_endpoint, input_df)
torch_preds <- predict(torch_endpoint, input_df)

# Convert torch logits to probabilities
torch_probs <- torch_softmax(torch_tensor(torch_preds), dim = 2)

# Compare
cor(as.vector(keras_preds), as.vector(as_array(torch_probs)))
```
