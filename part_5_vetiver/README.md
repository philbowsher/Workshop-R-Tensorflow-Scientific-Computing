# Part 5: Modern MLOps with Vetiver

This folder demonstrates **modern MLOps practices** using [Vetiver](https://vetiver.rstudio.com/), a framework from Posit for versioning, deploying, and monitoring models.

## What is Vetiver?

Vetiver is an open-source MLOps framework that:
- ✅ Versions models automatically
- ✅ Generates APIs without writing plumber.R manually
- ✅ Deploys to Connect programmatically
- ✅ Provides monitoring endpoints (/ping, /metadata)
- ✅ Works with multiple frameworks (Keras, Torch, tidymodels, etc.)

Think of it as **"DevOps best practices for ML models"**

## Quick Comparison: Manual vs Vetiver

### Manual Approach (Parts 3 & 4)

**Step 1: Pin model**
```r
pins::pin_upload(board, "model.keras", name = "my_model")
```

**Step 2: Write plumber.R manually**
```r
#* @get /predict
function(peptide) {
  # Load model
  # Encode input
  # Make prediction
  # Return result
}
```

**Step 3: Deploy**
```r
rsconnect::deployApp(appFiles = c("plumber.R", "requirements.txt"))
```

**Total files:** 3+ (model, plumber.R, requirements.txt, deploy script)

---

### Vetiver Approach (Part 5)

**Step 1: Create vetiver model**
```r
v <- vetiver_model(model, "my_model")
```

**Step 2: Pin with Vetiver**
```r
vetiver_pin_write(board, v)
```

**Step 3: Deploy (API auto-generated!)**
```r
vetiver_deploy_rsconnect(board, "my_model")
```

**Total files:** 1 model file
**plumber.R:** Auto-generated by Vetiver!

## Key Benefits of Vetiver

### 1. No Manual API Code
**Manual (Part 3):**
```r
# You write this entire plumber.R file:
library(plumber)
library(keras)
library(pins)

con <- pins::board_connect()
mod_path <- pins::pin_download(con, "model")
mod <- keras$saving$load_model(mod_path)

#* @get /predict
function(peptide) {
  # 50+ lines of encoding, prediction, error handling...
}
```

**Vetiver (Part 5):**
```r
# Vetiver generates everything automatically!
vetiver_deploy_rsconnect(board, "model_name")
```

---

### 2. Built-in Monitoring

**Manual (Part 3):**
- No health check endpoint
- No metadata endpoint
- You have to build these yourself

**Vetiver (Part 5):**
- ✅ `/ping` - Health check (is API alive?)
- ✅ `/metadata` - Model info (version, framework, description)
- ✅ `/predict` - Predictions (auto-generated)

---

### 3. Automatic Versioning

**Manual (Part 3):**
```r
# You track versions yourself
pins::pin_upload(board, "model_v1.keras", name = "model")
pins::pin_upload(board, "model_v2.keras", name = "model")
# How do you know which version is deployed?
```

**Vetiver (Part 5):**
```r
# Vetiver tracks versions automatically
vetiver_pin_write(board, v)  # Version 1
# Make changes
vetiver_pin_write(board, v)  # Version 2
# Versions are tracked, can rollback easily
```

## API Differences

### Manual Plumber API (Parts 3 & 4):
```bash
# GET request with query parameters
GET /predict?peptide=LLTDAQRIV

# Response
{"peptide": "LLTDAQRIV", "predicted_class": "WB"}
```

### Vetiver API (Part 5):
```bash
# POST request with JSON body (REST standard)
POST /predict
Body: {"V1": 0.5, "V2": 0.3, ..., "V180": 0.1}

# Response
[0.1, 0.7, 0.2]  # Probabilities for [NB, WB, SB]

# Additional endpoints
GET /ping       # Health check
GET /metadata   # Model info
```

## Production Considerations

Vetiver provides production-ready features:

1. **Model Cards**: Automatic documentation
2. **Versioning**: Track model changes over time
3. **Monitoring**: /ping endpoint for health checks
4. **Metadata**: /metadata endpoint for model info
5. **Standardization**: Same API structure across all models
6. **Rollback**: Easy to revert to previous versions
